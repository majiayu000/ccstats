# Product Spec

## Linked Issue

GH-32

## 用户问题

pricing 缓存写入会先截断 `pricing.json`，随后流式写入并静默吞掉所有错误。进程中断、磁盘满或并发运行可能留下损坏缓存；离线模式读到损坏缓存后会静默落到空定价表，用户只看到所有成本为 `N/A`。

## 目标

- pricing 缓存写入必须以原子替换完成，不能留下被截断的目标文件。
- 缓存写入失败必须至少向 stderr 发出可见 warning。
- 离线模式读到损坏缓存时必须 fail closed，返回明确错误，不能静默显示全表 `N/A`。
- 缓存缺失与缓存损坏必须是可区分状态。

## 非目标

- 不改变 LiteLLM 定价解析、模型匹配或 fallback pricing 规则。
- 不引入新的外部缓存服务。
- 不改变在线模式成功拉取定价后的用户输出格式。

## Behavior Invariants

1. 当缓存写入成功时，目标 `pricing.json` 只能从旧完整文件原子切换到新完整文件。
2. 当写入临时文件、flush、sync 或 rename 失败时，现有 `pricing.json` 不得被截断或替换为部分内容。
3. 当保存缓存失败时，正常在线统计仍可继续使用刚拉取的定价数据，但 stderr 必须包含 cache write warning。
4. 当 `--offline` 读取到缺失缓存时，行为保持为无缓存的离线路径。
5. 当 `--offline` 读取到损坏 JSON 或无法反序列化的缓存时，命令必须返回错误或显式错误状态，不能将成本静默降级为全 `N/A`。
6. statusline 的 quiet 加载路径不得吞掉会导致用户看到错误成本的缓存损坏。

## 验收标准

- [ ] 写缓存使用同目录临时文件加 `fs::rename` 原子替换。
- [ ] 写缓存失败不会破坏已有完整缓存，并会产生 warning。
- [ ] 离线读取损坏缓存会失败并展示明确错误。
- [ ] 缓存不存在与缓存损坏的行为有测试覆盖。
- [ ] 现有在线拉取、缓存命中和定价计算测试继续通过。

## 边界情况

- 目标缓存目录不存在。
- 临时文件创建失败。
- JSON 写入失败。
- flush 或 sync 失败。
- rename 失败。
- 缓存文件存在但内容为空、截断或非 JSON object。
- 两个 ccstats 进程并发保存缓存。

## 发布说明

这是 U-29 类静默降级修复。用户可见变化是损坏 pricing cache 在离线模式下会明确报错，而不是输出误导性的 `N/A` 成本。
